## PR-1: Relay Runtime Foundation + Safety Timeout

### Problem
- Four parallel global arrays (`gRelayActivationTime[]`, `gRelayActiveForMonitor[]`, `gRelayActiveMaskForMonitor[]`) tracked relay state inconsistently, making ownership and timeout logic fragile.
- No safety timeout for `MANUAL_RESET` mode — a stuck relay could stay ON indefinitely.

### Changes

**New types:**
- `RelaySource` enum: `RELAY_SRC_NONE`, `RELAY_SRC_ALARM`, `RELAY_SRC_MANUAL`, `RELAY_SRC_CLEAR_BUTTON`
- `RelayRuntime` struct: `{ bool active; uint8_t ownerMonitor; RelaySource source; unsigned long activatedAt; uint32_t customDurationSec; }`
- `gRelayRuntime[MAX_RELAYS]` replaces all three old global arrays

**New config field:**
- `MonitorConfig.relayMaxOnSeconds` (uint32_t, default 0 = no limit, max 604800 = 7 days)
- Parsed from config JSON, serialized back; `CONFIG_SCHEMA_VERSION` bumped to 2

**New helper functions:**
- `findMonitorForRelay(relayIndex)` — resolves which monitor owns a relay bit
- `activateRelayForMonitor(monitorIdx, relayMask, source, timestamp)` — unified activation
- `deactivateRelayForMonitor(monitorIdx, relayMask)` — unified deactivation with cleanup
- `isMonitorRelayActive(monitorIdx)` — replaces `gRelayActiveForMonitor[i]` checks
- `getMonitorActiveRelayMask(monitorIdx)` — replaces `gRelayActiveMaskForMonitor[i]` reads

**Refactored functions:**
- `sendAlarm()` — relay activation/deactivation uses new helpers
- `processRelayCommand()` — duration parameter support via `customDurationSec`
- `checkRelayMomentaryTimeout()` — complete rewrite: iterates per-relay, checks both momentary timeout AND safety `relayMaxOnSeconds` / `customDurationSec`, sends `relay_timeout` alarm when safety fires
- `resetRelayForMonitor()` — simplified to use helpers
- `setup()`, config change handler, hibernate recovery, `clearAllRelayAlarms()` — all updated

**Server changes:**
- Config Generator UI: new "Safety Max ON Duration" field (visible in `manual_reset` mode)
- `collectConfig()` / `loadConfig()` JS: `relayMaxOnSeconds` round-trip
- `handleAlarm()`: `relay_timeout` alarm type → clears `alarmActive`, sends SMS via clear policy
- `removeRelaySection()`: resets max-on field

**Relay overlap validation:**
- After parsing monitor configs, logs a warning if two monitors share relay bits

## PR-2: Debounce Counter Split

### Problem
A single `clearDebounceCount` was shared between high-alarm-clear and low-alarm-clear paths. When both thresholds were active (analog sensors), clearing one alarm could reset the debounce progress of the other.

### Changes
- `MonitorRuntime.clearDebounceCount` → split into `highClearDebounceCount` + `lowClearDebounceCount`
- **Digital alarm path:** Uses `highClearDebounceCount` only (digital sensors have a single alarm condition)
- **Analog alarm path:** High-alarm clear uses `highClearDebounceCount`; low-alarm clear uses `lowClearDebounceCount`. Removed shared `if (highCondition || lowCondition) clearDebounceCount = 0` guard; each condition now independently resets only its own counter.
- All initialization sites (setup, power recovery, config change) updated to zero both counters

## PR-4: DFU Integrity Verification (CRC-32)

### Problem
`tankalarm_performIapUpdate()` wrote firmware to flash and rebooted with zero verification. A bit flip, power glitch, or I2C corruption during the write loop would cause a permanent brick (JTAG recovery required in the field).

### Changes
- Added `tankalarm_crc32()` — ISO 3309 CRC-32 (same polynomial as zlib/PNG), incremental computation
- **During download:** Running CRC-32 is accumulated over each decoded chunk's raw bytes (before page-alignment padding)
- **After write loop:** Reads back the entire written flash region via memory-mapped pointer, computes read-back CRC-32 in 4KB chunks (with watchdog kicks)
- **Verification:** If download CRC ≠ flash CRC, prints mismatch, does NOT reboot, restores hub mode, returns false
- **On match:** Logs "CRC verified OK", proceeds with normal reboot

Both CRC values are printed to Serial for diagnostics.

## Migration Notes

- **Config schema 1 → 2:** Automatic. Missing `relayMaxOnSeconds` field defaults to 0 (no limit). No manual intervention needed.
- **Relay state:** After upgrading, relay tracking state is re-initialized (all relays start inactive). Active alarms will re-trigger relays on the next evaluation cycle.
- **DFU:** The CRC verification adds ~2-5 seconds to the update process (flash read-back time). No user-facing change on success.
